1. Visão Geral do Sistema

O TaskManager é uma aplicação web full-stack para gerenciamento de tarefas e usuários. Permite criar, listar, atualizar e deletar tarefas e usuários, com persistência de dados em banco de dados na nuvem.

O que o sistema faz

O sistema possui duas entidades principais: Usuários e Tarefas. Para cada uma delas é possível realizar as quatro operações básicas de um sistema CRUD:

Criar
Cadastrar novo usuário ou tarefa
Listar
Visualizar todos os registros
Atualizar
Editar dados ou alterar status
Deletar
Remover um registro pelo ID

Arquitetura geral

O projeto é dividido em dois serviços independentes hospedados na Vercel:

Frontend — Páginas HTML/CSS/JS estáticas servidas pela Vercel.

Backend — API REST em Node.js/Express, também hospedada na Vercel como função serverless. Conecta-se ao Supabase para persistência dos dados.

3. Arquitetura em Camadas

O backend segue o padrão de arquitetura em camadas (Layered Architecture), onde cada camada tem uma única responsabilidade e só se comunica com a camada imediatamente abaixo.

1
Routes (Rotas)
src/routes/tarefaRoutes.js  |  src/routes/usuarioRoutes.js

É a porta de entrada da API. Define quais endereços (URLs) existem e qual método HTTP cada um aceita (GET, POST, PUT, DELETE). Quando uma requisição chega, a rota direciona para o controller correto. Não contém nenhuma lógica — só mapeia URL para função.

2
Controllers (Controladores)
src/controllers/tarefaController.js  |  src/controllers/usuarioController.js

Recebe o objeto de requisição HTTP (req) e monta a resposta (res). Extrai os dados que vieram do corpo da requisição ou da URL, repassa para o service, e devolve o resultado como JSON. Também trata erros e define os códigos HTTP de resposta (200, 201, 500, etc.).

3
Services (Serviços)
src/services/tarefaService.js  |  src/services/usuarioService.js

É onde ficam as regras de negócio. Por exemplo: ao criar uma tarefa, se o campo status não for informado, o service define automaticamente como "pendente". O service não sabe nada sobre HTTP — ele só recebe dados e devolve resultados, chamando o repository para operações de banco.

4
Repositories (Repositórios)
src/repositories/tarefaRepository.js  |  src/repositories/usuarioRepository.js

Única camada que se comunica diretamente com o banco de dados (Supabase). Encapsula todas as operações: select, insert, update e delete. Se o banco de dados for trocado no futuro, só essa camada precisa ser alterada.

5
Models (Modelos)
src/models/tarefa.js  |  src/models/usuario.js

Define a estrutura das entidades do sistema. A classe Tarefa possui os campos: id, titulo, descricao e status. A classe Usuario possui: id, nome e email. Cada model também exporta uma função factory para criar instâncias.

6
Database — Singleton
src/database/singleton.js

Cria uma única instância de conexão com o Supabase que é compartilhada em todo o projeto. O padrão Singleton garante que não sejam abertas múltiplas conexões desnecessárias. Utiliza as variáveis de ambiente SUPABASE_URL e SUPABASE_KEY para autenticar.

4. Fluxo de uma Requisição

Veja como um dado percorre todo o sistema, desde o clique do usuário no navegador até o retorno do banco de dados.

Exemplo: criar uma nova tarefa

Navegador
clica em "Criar Tarefa"
→
script.js
POST /tarefas
→
tarefaRoutes
direciona
→
Controller
extrai body
Service
aplica regras
→
Repository
INSERT no banco
→
Supabase
salva e retorna
1. Navegador — O usuário preenche o formulário e clica no botão. O script.js faz uma requisição POST para /tarefas com os dados em formato JSON.

2. Routes — O Express identifica que POST /tarefas deve ser tratado pela função tarefaController.criar.

3. Controller — Lê os dados do req.body, chama tarefaService.criar() e aguarda o resultado.

4. Service — Verifica se o status foi informado; se não, define como "pendente". Chama tarefaRepository.salvar().

5. Repository — Executa o INSERT na tabela tarefas do Supabase e retorna o registro criado.

6. Resposta — O dado percorre o caminho inverso até chegar ao controller, que responde com status 201 e o JSON da tarefa criada. O frontend atualiza a lista na tela.

Detecção de ambiente no Frontend

O script.js detecta automaticamente se está rodando em ambiente local ou em produção:

const API_URL = window.location.hostname === "localhost" ? "http://localhost:3000" // ambiente local : "https://arquitetura-de-sistemas.vercel.app"; // produção

5. API — Endpoints de Tarefas

Todos os endpoints disponíveis para operações com tarefas.

Base URL (produção): https://arquitetura-de-sistemas.vercel.app

Base URL (local): http://localhost:3000

MétodoEndpointDescriçãoBodyRetorno
GET /tarefas Lista todas as tarefas — Array de tarefas
POST /tarefas Cria uma nova tarefa titulo, descricao, status Tarefa criada (201)
PUT /tarefas/:id Atualiza uma tarefa titulo, descricao, status Tarefa atualizada
DELETE /tarefas/:id Remove uma tarefa — { mensagem: "Removida com sucesso" }

Exemplo — Criar tarefa (POST /tarefas)

Requisição:

POST /tarefas Content-Type: application/json { "titulo": "Revisar relatório", "descricao": "Revisar o relatório mensal de vendas", "status": "pendente" }

Resposta (201 Created):

{ "id": 1, "titulo": "Revisar relatório", "descricao": "Revisar o relatório mensal de vendas", "status": "pendente" }

Status possíveis de uma tarefa

ValorDescriçãoCor na interface
pendenteTarefa ainda não iniciadaAmarelo
em andamentoTarefa em progressoAzul
concluidaTarefa finalizadaVerde

6. API — Endpoints de Usuários

Todos os endpoints disponíveis para operações com usuários.

MétodoEndpointDescriçãoBodyRetorno

Exemplo — Criar usuário (POST /usuarios)

Requisição:

POST /usuarios Content-Type: application/json { "nome": "João Silva", "email": "joao@email.com" }

Resposta (201 Created):

{ "id": 1, "nome": "João Silva", "email": "joao@email.com" }

Tratamento de erros

Quando ocorre um erro em qualquer operação, a API retorna status 500 com a mensagem do erro:

{ "erro": "mensagem descrevendo o problema" }

8. Banco de Dados — Supabase

O Supabase é um serviço de banco de dados PostgreSQL na nuvem com autenticação via chave de API.

Conexão — Padrão Singleton

O arquivo singleton.js cria uma única instância do cliente Supabase que é compartilhada em todo o projeto:

const { createClient } = require('@supabase/supabase-js'); const supabase = createClient( process.env.SUPABASE_URL, // URL do projeto process.env.SUPABASE_KEY // chave anon public ); module.exports = supabase;

Operações no banco

OperaçãoCódigo SupabaseSQL equivalente
Listar tudo .from('tarefas').select('*') SELECT * FROM tarefas
Inserir .from('tarefas').insert(dados) INSERT INTO tarefas ...
Atualizar .from('tarefas').update(dados).eq('id', id) UPDATE tarefas SET ... WHERE id = ?
Deletar .from('tarefas').delete().eq('id', id) DELETE FROM tarefas WHERE id = ?

Tabelas necessárias no Supabase

O projeto espera que duas tabelas existam no banco:

-- Tabela de tarefas CREATE TABLE tarefas ( id SERIAL PRIMARY KEY, titulo TEXT NOT NULL, descricao TEXT NOT NULL, status TEXT DEFAULT 'pendente' ); -- Tabela de usuários CREATE TABLE usuarios ( id SERIAL PRIMARY KEY, nome TEXT NOT NULL, email TEXT NOT NULL );
Onde criar as tabelas: No painel do Supabase, acesse SQL Editor e execute os comandos acima para criar as tabelas.

10. Variáveis de Ambiente

Variáveis de ambiente são configurações sensíveis que ficam fora do código-fonte. No ambiente local ficam no arquivo .env; em produção são configuradas no painel da Vercel.

Variáveis necessárias

VariávelDescriçãoOnde encontrar
SUPABASE_URL URL do projeto no Supabase Supabase → Project Settings → API → Project URL
SUPABASE_KEY Chave de acesso anon/public Supabase → Project Settings → API → anon public
PORT Porta do servidor local Definida manualmente (padrão: 3000)

Arquivo .env (ambiente local)

SUPABASE_URL=https://seuprojeto.supabase.co SUPABASE_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... PORT=3000
Segurança: O arquivo .env está listado no .gitignore, o que significa que ele nunca é enviado ao GitHub. Cada desenvolvedor precisa criar o seu próprio arquivo .env com as credenciais corretas.

Como configurar na Vercel (produção)

Para que o backend em produção funcione, as mesmas variáveis precisam ser cadastradas no painel da Vercel:

1. Acesse vercel.com → seu projeto de backend
2. Vá em Settings → Environment Variables
3. Adicione SUPABASE_URL e SUPABASE_KEY com os mesmos valores do .env
4. Clique em Save e depois em Redeploy

11. Deploy — Vercel

O projeto usa dois deployments separados na Vercel: um para o backend (API) e um para o frontend.

Backend — vercel.json

O arquivo backend/vercel.json instrui a Vercel a tratar o server.js como uma função Node.js e redirecionar todas as rotas para ele:

{ "version": 2, "builds": [ { "src": "server.js", "use": "@vercel/node" } ], "routes": [ { "src": "/(.*)", "dest": "/server.js" } ] }

Informações de produção

ServiçoURLPlataforma

Frontend URL do projeto frontend na Vercel Vercel (Static)
Banco de dados pfcwizxvepuklbrogrtf.supabase.co Supabase (PostgreSQL)

Como funciona o deploy automático

A Vercel está conectada ao repositório GitHub. A cada git push na branch main, ela detecta as mudanças e faz um novo deploy automaticamente — tanto do backend quanto do frontend.

12. Como Rodar Localmente

Passo a passo para executar o projeto na sua máquina.

Pré-requisitos

Node.js
v18 ou superior
npm
Incluído com o Node.js
Conta Supabase
Com projeto criado e tabelas configuradas
Arquivo .env
Com SUPABASE_URL e SUPABASE_KEY

Passo 1 — Clonar o repositório

git clone https://github.com/seu-usuario/seu-repositorio.git cd projeto---arquitetura-de-sistemas

Passo 2 — Instalar dependências do backend

cd sistema-tarefas/backend npm install

Passo 3 — Criar o arquivo .env

Crie o arquivo sistema-tarefas/backend/.env com o seguinte conteúdo:

SUPABASE_URL=https://seuprojeto.supabase.co SUPABASE_KEY=sua_chave_anon_public PORT=3000

Passo 4 — Iniciar o servidor

npm run dev # Saída esperada: Servidor rodando na porta 3000

Passo 5 — Abrir o frontend

Abra o arquivo sistema-tarefas/frontend/index.html diretamente no navegador. O script.js detecta automaticamente que está em localhost e direciona as requisições para http://localhost:3000.

Pronto! O sistema estará funcional localmente. Qualquer alteração salva no código é refletida ao recarregar o navegador. Para parar o servidor, pressione Ctrl + C no terminal.
